# shared-inspector Demo Repository Plan

A practical implementation guide for a controlled, realistic demo repo that demonstrates
`@mf-toolkit/shared-inspector` against a microfrontend e-commerce platform.

---

## Repository Concept

**Name:** `mf-storefront-demo`

A microfrontend e-commerce storefront split into three independently deployable apps that
share a common shell (host). The architecture mirrors real-world setups where teams own
separate domains and drift naturally occurs over time.

**Apps:**
| App | Role | Domain |
|---|---|---|
| `shell` | Host / container | Routing, layout, global auth state |
| `catalog` | Remote | Product listing, search, filters |
| `checkout` | Remote | Cart, order summary, payment flow |

Each scenario is a separate Git branch or tagged commit in the demo repo. The `healthy`
state is `main`; `drift` and `federation-issues` are separate branches with surgical
changes applied on top.

**Tool being demonstrated:**
- Per-app analysis: `npx mf-inspector --source src --shared shared-config.json`
- Cross-app federation analysis: `npx mf-inspector federation shell/project-manifest.json catalog/project-manifest.json checkout/project-manifest.json`

## Shared Dependency Matrix

This table shows which packages each app **uses in source code** vs. which it **declares in
its `shared` config**. The matrix is the ground truth for what the inspector should find
in each scenario.

### Healthy baseline (Scenario 1)

| Package | shell uses | shell shared | catalog uses | catalog shared | checkout uses | checkout shared |
|---|---|---|---|---|---|---|
| `react` | yes | yes (singleton) | yes | yes (singleton) | yes | yes (singleton) |
| `react-dom` | yes | yes (singleton) | yes | yes (singleton) | yes | yes (singleton) |
| `react-router-dom` | yes | yes (singleton) | yes | yes (singleton) | yes | yes (singleton) |
| `zustand` | yes | yes (singleton) | no | no | yes | yes (singleton) |
| `lodash` | no | no | yes | no | no | no |

**Notes:**
- `lodash` is intentionally not shared — it is a utility belt, not a stateful runtime, so
  each MF bundling its own copy is acceptable and correct.
- `zustand` is only used by `shell` and `checkout` (cross-MF cart/auth state), both
  correctly declare it as a singleton.
- `lodash` appears as a share candidate in the inspector output, but the developer
  consciously accepts this (LOW severity, easy to dismiss).

### Drift state (Scenario 2) — catalog drift + checkout eager risk

| App | Package | Issue | Category |
|---|---|---|---|
| `catalog` | `react` | declared `^17.0.2`, installed `18.3.1` | mismatch (HIGH) |
| `catalog` | `react-dom` | declared `^17.0.2`, installed `18.3.1` | mismatch (HIGH) |
| `catalog` | `zustand` | declared without `singleton: true` | singletonRisk (MEDIUM) |
| `catalog` | `date-fns` | declared, 0 imports | unused (LOW) |
| `checkout` | `react-router-dom` | `eager: true` without `singleton: true` | eagerRisk (MEDIUM) |

### Federation issues (Scenario 3) — cross-MF config divergence

| Package | shell shared | catalog shared | checkout shared | issue |
|---|---|---|---|---|
| `react-router-dom` | `^6.22.0` singleton | `^6.8.0` singleton | `^6.22.0` singleton | version conflict (HIGH) |
| `zustand` | singleton: true | singleton: true | singleton: false | singleton mismatch (MEDIUM) |
| `lodash` | yes (shared) | no | no | ghost share (LOW) |
| `date-fns` | no | no | no | host gap — used by checkout, unshared by all (MEDIUM) |

## Scenario 2: Drift

### What it demonstrates

A remote (`catalog`) whose shared config has silently drifted from the rest of the
federation. This models real-world config rot: a developer upgraded React in `catalog` but
forgot to update the `requiredVersion` in the shared config. They also added `zustand` to
the shared config without `singleton: true` (copy-paste from a non-stateful package) and
left a now-unused `date-fns` in the config from a previous feature that was removed.

The inspector catches all three classes of problem in a single run against `catalog` alone.

### Intentional problems introduced

1. **Version mismatch — `react` and `react-dom` (HIGH, -20 each)**
   - `catalog/package.json`: keep `react` and `react-dom` at `18.3.1` (installed)
   - `catalog/shared-config.json`: set `requiredVersion` to `^17.0.2`
   - This simulates a developer who copied a shared config from an old project

2. **Singleton risk — `zustand` (MEDIUM, -8)**
   - Add `zustand` to `catalog`'s dependencies: `"zustand": "4.5.2"`
   - Add `zustand` to `catalog/shared-config.json` **without** `singleton: true`:
     `"zustand": { "requiredVersion": "^4.5.2" }`
   - `catalog/src/` does NOT import `zustand` (the shared declaration is preemptive/wrong)
   - Result: also triggers an **unused shared** finding for `zustand`

3. **Unused shared — `date-fns` (LOW, -3)**
   - Add `"date-fns"` to `catalog/shared-config.json`: `"date-fns": { "singleton": false }`
   - Do NOT import `date-fns` anywhere in `catalog/src/`
   - `date-fns` must be present in `catalog/package.json` dependencies so version
     resolution can run

4. **Eager risk — `react-router-dom` in `checkout` (MEDIUM, -8)**
   - In `checkout/shared-config.json`: change `react-router-dom` to
     `{ "eager": true, "requiredVersion": "^6.22.3" }` — remove `singleton: true`
   - `eager: true` without `singleton: true` means the package is loaded immediately at
     startup but not guaranteed to be a single instance; if another MF also loads it
     eagerly, two separate router instances can exist
   - This is independent of the catalog drift — demonstrates that per-app issues can
     exist in multiple MFs simultaneously, each caught by the inspector separately

```json
  "zustand":          { "requiredVersion": "^4.5.2" },
  "date-fns":         {}
}
```

## Scenario 3: Federation Issues

### What it demonstrates

Problems that are invisible when analyzing each MF in isolation, but immediately apparent
when the inspector runs in `federation` mode across all three manifests. This models the
class of bugs that only appear in production after all three apps are deployed together.

### Intentional problems introduced

Each change is applied to the shared config only — the source code stays identical to
the healthy baseline.

1. **Version conflict — `react-router-dom` (HIGH, -20)**
   - `catalog/shared-config.json`: change `requiredVersion` to `"^6.8.0"`
   - `shell` and `checkout` remain at `"^6.22.3"`
   - MF singleton negotiation picks one version; the MF that loses gets a router with
     a different API surface (`useNavigate` behavior changed between 6.8 and 6.22)

2. **Singleton mismatch — `zustand` (MEDIUM, -8)**
   - `checkout/shared-config.json`: change zustand entry to `{ "requiredVersion": "^4.5.2" }`
     (remove `singleton: true`)
   - `shell` retains `singleton: true`
   - Result: the `authStore` in `shell` and `cartStore` in `checkout` may end up as
     separate Zustand instances depending on which MF loads first

3. **Ghost share — `lodash` (LOW, -3)**
   - Add `lodash` to `shell/shared-config.json`: `"lodash": {}`
   - `shell/src/` does not import `lodash`
   - `catalog` uses `lodash` but does not declare it in shared
   - The inspector sees `lodash` shared by `shell` only, used unshared by `catalog`
   - This is one-sided coupling: `shell` declares it shared but gets no federation benefit

4. **Host gap — `date-fns` (MEDIUM, -8)**
   - Add `date-fns` usage to `checkout/src/Cart.tsx`:
     ```typescript
     import { format } from 'date-fns';
     ```
   - Do NOT add `date-fns` to any app's shared config
   - All three MFs bundle their own copy (only `checkout` uses it, but the inspector
     flags it as a host gap because it is not declared shared by anyone)

#### Webpack plugin alternative (catalog)

Instead of running the CLI manually, `catalog` can integrate the inspector directly into
its webpack build. This is the recommended setup for teams that want CI to catch drift
automatically on every build.

**catalog/webpack.config.js**
```typescript
const { ModuleFederationPlugin } = require('webpack').container;
const { MfSharedInspectorPlugin } = require('@mf-toolkit/shared-inspector/webpack');

module.exports = {
  plugins: [
    new ModuleFederationPlugin({
      name: 'mf-storefront-catalog',
      shared: require('./shared-config.json'),
    }),

    new MfSharedInspectorPlugin({
      sourceDirs: ['./src'],
      // sharedConfig auto-extracted from ModuleFederationPlugin above
      failOn: 'mismatch',      // exit 1 when any version mismatch is detected
      warn: true,              // print findings to console even without failOn
      writeManifest: true,     // emit project-manifest.json for federation analysis
      outputDir: '.',
    }),
  ],
};
```

In drift state, the webpack build exits with code 1 and prints:

```
[MfSharedInspector] mf-storefront-catalog — build failed (failOn: mismatch)

⚠  Version Mismatch — react     configured: ^17.0.2 | installed: 18.3.1
⚠  Version Mismatch — react-dom configured: ^17.0.2 | installed: 18.3.1

2 mismatch(es) found. Exiting with code 1.
```

> **Demo note:** Same findings as the CLI run, but the build itself fails. CI never
> produces a deployable artifact. The team is blocked until the config is fixed —
> this is the intended behaviour for HIGH-severity drift.

#### Per-app analysis (each app individually)

Running `mf-inspector` against `shell`, `catalog`, and `checkout` individually in

```
[MfSharedInspector] federation analysis (3 MFs)
```
